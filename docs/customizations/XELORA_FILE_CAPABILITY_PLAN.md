# Xelora 文件能力建设规划 v1

本文档用于指导 Xelora 在网页端补齐“文件生成、文件修改、文件预览、文件落盘”能力，重点覆盖 Markdown、Excel/CSV、PPT、PDF 四类高频项目交付物。

目标不是一次性做成完整在线 Office，而是先把智能体可稳定调用、结果可真实产出、前端可直接预览这三件事做扎实，再逐步扩展更复杂的协作与模板场景。

## 1. 背景与问题

当前 Xelora 已具备知识库问答、智能体、Skill 扩展和 Docker 化部署能力，但在网页端文件处理链路上仍有明显缺口：

- 智能体可以“理解”文件生成意图，但不一定能稳定生成真实文件。
- Skill 调用在网页端存在参数传递、脚本执行、结果回传不闭环的问题。
- 文件输出缺少统一的工作区、元数据和预览入口。
- 高频业务场景如项目汇报、报告整理、表格清洗、知识库导出尚未产品化。

这些问题会直接影响 Xelora 是否能从“问答平台”升级为“可交付工作成果的平台”。

## 2. 建设目标

一期目标：

1. 支持生成 Markdown、Excel/CSV、PPT、PDF 文件。
2. 支持对已有 Markdown、Excel/CSV 文件做结构化修改并保存新版本。
3. 支持网页端查看本次会话实际产出的文件，而不是只显示文本结果。
4. 支持智能体稳定调用文件相关 Skill，执行结果可审计、可定位、可复现。

非一期目标：

- 不在一期建设完整多人协同编辑。
- 不在一期建设完整 Word 在线编辑器。
- 不追求任意脚本在网页端自由执行，优先收敛为平台级文件能力。

## 3. 设计原则

### 3.1 先平台能力，后 Skill 编排

文件生成与修改优先做成平台原生能力，Skill 负责调度和组织流程，而不是让每个 Skill 自己直接落文件。

### 3.2 先高频交付物，后全格式覆盖

优先覆盖项目交付中最常见的四类结果：

- Markdown 报告
- Excel/CSV 表格
- PPT 汇报材料
- PDF 导出件

### 3.3 结果必须真实落盘

“生成成功”的定义不是聊天窗口给出一段文本，而是：

- 生成了真实文件
- 文件进入统一输出目录
- 前端可以查看和下载
- 后端记录了文件元数据

### 3.4 Web 场景优先

考虑 Xelora 是网页端产品，不直接复制 IDE 的任意本地执行模型。优先建设：

- 可控输入
- 可控输出
- 可控权限
- 可审计执行链路

## 4. 总体架构

建议将文件能力拆成四层：

### 4.1 交互层

职责：

- 提供文件上传、生成、编辑、导出、预览入口
- 在对话页展示“本次输出文件”
- 在智能体页展示支持的文件能力

建议组件：

- Markdown 编辑与预览：Tiptap 或现有富文本组件增强
- 表格编辑：Univer
- PDF 预览：pdf.js

### 4.2 文件能力层

职责：

- 统一封装不同文件格式的生成、读取、修改、导出逻辑
- 为上层提供稳定接口，不暴露底层脚本细节

建议适配器：

- Markdown：文本模板 + 格式化流程
- Excel/CSV：SheetJS
- PPT：PptxGenJS
- PDF 导出：Gotenberg

### 4.3 Skill 执行层

职责：

- 将“读取 Skill / 执行 Skill”与平台文件能力打通
- 统一参数格式、超时、日志、输出文件清单

建议约束：

- `args` 必须标准化为字符串数组
- 输出结果必须包含 `exit_code`、`stdout`、`stderr`、`output_files`
- 脚本不得直接写未知路径，只能写入平台分配的工作目录

### 4.4 基础设施层

职责：

- 提供文件工作区、转换服务、容器运行环境和元数据存储

建议新增：

- `workspace/uploads/`：用户上传文件
- `workspace/outputs/`：生成结果
- `workspace/temp/`：中间产物
- `Gotenberg`：HTML/Markdown/Office -> PDF

## 5. 推荐技术选型

以下选型优先考虑成熟度、社区活跃度、与 Web 场景的适配度：

| 能力 | 推荐方案 | 用途 |
|---|---|---|
| Markdown 编辑 | Tiptap | 报告整理、富文本编辑、结构化内容生成 |
| Excel/CSV 读写 | SheetJS | 导入、修改、导出 xlsx/csv |
| 在线表格编辑 | Univer | 网页端表格查看与编辑 |
| PPT 生成 | PptxGenJS | 基于内容或模板生成项目汇报 |
| PDF 导出 | Gotenberg | 报告与页面输出 PDF |
| PDF 预览 | pdf.js | 网页端查看 PDF |
| 通用转换补充 | Pandoc | Markdown/HTML/文档之间的补充转换 |

说明：

- `ONLYOFFICE` 很强，但接入、资源占用和许可证治理更重，不建议作为一期默认方案。
- 一期应优先选“轻集成、低耦合、便于 Docker 部署”的组件组合。

## 6. 一期能力范围

### 6.1 Markdown 工作流

支持：

- 根据用户指令生成 `report.md`
- 对生成内容进行格式化
- 前端预览 Markdown
- 将 Markdown 导出为 PDF

典型场景：

- 项目周报
- 知识库内容摘要
- 调研报告初稿

### 6.2 Excel/CSV 工作流

支持：

- 上传 `xlsx/csv`
- 读取表头和数据
- 智能体执行清洗、汇总、补列、公式填充
- 保存为新文件并回传下载地址

典型场景：

- 数据清洗
- 报名表整理
- 项目数据汇总

### 6.3 PPT 工作流

支持：

- 根据主题、知识库内容或 Markdown 生成 `pptx`
- 使用固定模板替换封面、目录、要点页内容

典型场景：

- 项目汇报
- 方案介绍
- 内部评审材料

### 6.4 PDF 工作流

支持：

- 将 Markdown/HTML 报告导出为 PDF
- 将生成后的结果纳入统一文件面板

一期暂不优先支持复杂 PDF 编辑，优先支持“生成”和“重新导出”。

## 7. 关键模块改造建议

### 7.1 新增 file-service

建议职责：

- 接收文件生成/修改请求
- 调用对应适配器
- 写入统一输出目录
- 记录文件元数据
- 返回文件访问路径

建议接口示例：

- `POST /api/files/markdown/generate`
- `POST /api/files/excel/transform`
- `POST /api/files/ppt/generate`
- `POST /api/files/pdf/export`
- `GET /api/files/:id`
- `GET /api/files/:id/download`

### 7.2 改造 Skill 执行结果结构

建议后端统一返回：

```json
{
  "exit_code": 0,
  "stdout": "",
  "stderr": "",
  "output_files": [
    {
      "name": "report.md",
      "path": "workspace/outputs/session-001/report.md",
      "mime_type": "text/markdown"
    }
  ]
}
```

收益：

- 前端能明确知道“生成了什么文件”
- 不再依赖从自然语言回复里猜测结果
- 失败时能明确是参数问题、脚本问题还是权限问题

### 7.3 新增文件结果面板

建议前端在对话页增加：

- 本次输出文件列表
- 预览按钮
- 下载按钮
- 重新生成按钮

对于表格和 PDF，可直接接入内嵌预览。

## 8. Docker 部署改造建议

在现有 Docker Compose 基础上，建议新增或扩展以下能力：

### 8.1 新增 Gotenberg 服务

用途：

- 将 HTML、Markdown 或 Office 文档转换为 PDF

接入方式：

- 作为独立容器加入 `docker-compose.yml`
- `app` 或 `file-service` 通过 HTTP 调用

### 8.2 挂载统一工作区

建议：

- 为 `app`、`sandbox`、`file-service` 挂载同一工作区根目录
- 明确区分上传目录、输出目录、临时目录

收益：

- 解决脚本写入路径不一致
- 便于前端下载、清理和追踪产物

## 9. 风险与控制

### 9.1 Skill 任意写文件风险

控制建议：

- Skill 只能写平台分配的输出目录
- 路径校验必须在后端完成

### 9.2 网页端执行链路不透明

控制建议：

- 保存每次执行的参数、日志、输出文件和退出码
- 提供最小可见的调试信息给前端

### 9.3 表格与文档格式复杂度失控

控制建议：

- 一期只支持高频结构化操作
- 复杂版式和深度 Office 兼容留到二期评估

## 10. 分阶段里程碑

### Phase 1：可落盘的文件输出闭环

目标：

- Markdown 生成并保存
- Skill 输出文件可被前端识别
- 对话页显示文件结果

验收标准：

- 用户在网页端发起生成报告请求后，能下载真实 `md/pdf`

### Phase 2：表格处理闭环

目标：

- 上传 `xlsx/csv`
- 智能体修改表格并保存新版本
- 前端可预览表格结果

验收标准：

- 用户能在网页端完成一次“上传表格 -> 修改 -> 下载结果”

### Phase 3：PPT 报告生成

目标：

- 基于模板生成 `pptx`
- 支持项目汇报类场景

验收标准：

- 用户能基于知识库或输入主题下载一份可打开的 PPT

### Phase 4：统一文件工作台

目标：

- 对话生成文件统一归档
- 支持再次编辑、二次导出和按会话追踪

验收标准：

- 用户能在平台内查看和管理历史生成文件

## 11. 建议优先级

建议按以下顺序推进：

1. 先修通 Skill 执行结果与文件落盘闭环
2. 再做 Markdown/PDF 输出
3. 然后做 Excel/CSV 修改
4. 最后做 PPT 模板生成和统一文件工作台

原因：

- 这条路径最接近用户当前真实需求
- 能最快从“回答内容”升级到“交付文件”
- 风险可控，不会因为一步到位引入过重系统复杂度

## 12. 近期实施清单

建议下一轮开发直接拆成以下可执行任务：

- T1：统一 Skill 执行结果结构，补全 `output_files`
- T2：新增工作区输出目录与路径校验
- T3：增加 Markdown 生成与保存接口
- T4：接入 Gotenberg，实现 Markdown/HTML 导出 PDF
- T5：对话页增加文件结果面板
- T6：接入 SheetJS，实现 Excel/CSV 导入导出
- T7：接入 Univer，实现基础表格预览与编辑
- T8：接入 PptxGenJS，实现模板化 PPT 输出

## 13. 结论

Xelora 下一阶段最值得投入的方向，不是继续增加“会回答”的能力，而是补齐“能产出文件、能修改文件、能让用户直接拿结果”的能力链路。

只要把文件能力沉淀为平台原生模块，Skill 就不再只是“脚本调用入口”，而会成为真正可编排的业务能力层。这样既能支持报告、表格、PPT、PDF 等高频项目场景，也能为后续知识库导出、模板化交付、自动化办公打好基础。
